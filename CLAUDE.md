# CLAUDE.md

このリポジトリで作業するときの運用ルール。

## サブエージェント自動連携フロー

ユーザーから**新機能・変更の要望**を受けたら、メインの Claude は以下を**全自動**でオーケストレーションする。各担当の成果物（ブランチ・仕様書・PR）は引き継ぎ情報で受け渡す。

サブエージェントは `.claude/agents/` に定義済み:
- `planner`（企画担当）— 仕様策定。ブランチ作成と仕様書コミットまで。
- `developer`（開発担当）— 実装・コミット・PR作成。
- `reviewer`（確認担当）— Playwright での動作確認・PRへの結果コメント。

### フロー

1. **planner を呼ぶ**
   - 要望をそのまま渡す。
   - 完了報告から「**ブランチ名**」「**仕様書のパス**」を抽出して保持する。
   - **例外的な停止条件**: planner が「未解決の確認事項」を返した場合のみ、判断材料が無いと先に進めないため、ここで停止してユーザーに確認する。確認が取れたらフローを再開する。これ以外では停止しない。

2. **developer を呼ぶ**
   - planner から抽出した「ブランチ名」「仕様書のパス」を明示して渡す。
   - 完了報告から「**PR の番号 / URL**」を抽出して保持する。

3. **reviewer を呼ぶ**
   - developer から抽出した「PR の番号」を渡す。
   - 完了報告から「**総合判定（マージ可 / 要修正）**」を抽出する。

4. **判定による分岐**
   - **マージ可** → フロー完了。3担当の成果（ブランチ名・仕様書パス・PR URL・確認結果の要約）をまとめてユーザーに報告する。
   - **要修正** → reviewer の**指摘内容を developer に渡して再実装**させ、**同じ PR** を再度 reviewer に確認させる。これを**最大 3 回**まで繰り返す。
     - 3 回繰り返してもマージ可にならない場合は、ループを止めてユーザーに報告する（無限ループ防止）。各回の指摘と対応の要約を残す。

### 原則

- 段階ごとのユーザー承認は挟まない（上記「例外的な停止条件」を除く）。
- 担当間の引き継ぎ情報（ブランチ名・仕様書パス・PR番号）は、必ず前段の**完了報告から抽出した実値**を次段に渡す。推測値で渡さない。
- 各担当の責務境界を尊重する: planner は実装しない、developer は受け入れ判定しない、reviewer はコード修正しない。
