# CLAUDE.md

このリポジトリで作業するときの運用ルール。

## サブエージェント自動連携フロー

ユーザーから**新機能・変更の要望**を受けたら、メインの Claude は以下を**全自動**でオーケストレーションする。各担当の成果物（ブランチ・**worktree**・仕様書・PR）は引き継ぎ情報で受け渡す。

実装作業は **git worktree で隔離する**。planner がブランチと同時に `.worktrees/<leaf>` を作成し、developer / reviewer はその worktree 内で作業する（メインのチェックアウトは `main` のまま汚さない）。`<leaf>` はブランチ名から `feature/` を除いた部分。詳細は各エージェント定義（`.claude/agents/`）と `superpowers:using-git-worktrees` スキルに従う。

サブエージェントは `.claude/agents/` に定義済み:
- `planner`（企画担当）— 仕様策定。ブランチ + worktree 作成と仕様書コミットまで。
- `developer`（開発担当）— 実装・コミット・PR作成。
- `reviewer`（確認担当）— Playwright での動作確認・PRへの結果コメント。

### フロー

1. **planner を呼ぶ**
   - 要望をそのまま渡す。
   - 完了報告から「**ブランチ名**」「**worktree のパス**」「**仕様書のパス**」を抽出して保持する。
   - **例外的な停止条件**: planner が「未解決の確認事項」を返した場合のみ、判断材料が無いと先に進めないため、ここで停止してユーザーに確認する。確認が取れたらフローを再開する。これ以外では停止しない。

2. **developer を呼ぶ**
   - planner から抽出した「ブランチ名」「worktree のパス」「仕様書のパス」を明示して渡す。
   - 完了報告から「**PR の番号 / URL**」を抽出して保持する。

3. **reviewer を呼ぶ**
   - developer から抽出した「PR の番号」と、planner の「worktree のパス」を渡す。
   - 完了報告から「**総合判定（マージ可 / 要修正）**」を抽出する。

4. **判定による分岐**
   - **マージ可** → **worktree を後片付けしてから**フロー完了。リポジトリ root で `git worktree remove <worktree パス>` を実行する（マージ後はブランチも不要なら削除可）。3担当の成果（ブランチ名・worktree パス・仕様書パス・PR URL・確認結果の要約）をまとめてユーザーに報告する。
   - **要修正** → reviewer の**指摘内容を developer に渡して再実装**させ、**同じ PR**（同じ worktree）を再度 reviewer に確認させる。これを**最大 3 回**まで繰り返す。
     - 3 回繰り返してもマージ可にならない場合は、ループを止めてユーザーに報告する（無限ループ防止）。各回の指摘と対応の要約を残す。worktree はユーザーが追って判断できるよう残しておく。

### 原則

- 段階ごとのユーザー承認は挟まない（上記「例外的な停止条件」を除く）。
- 担当間の引き継ぎ情報（ブランチ名・worktree パス・仕様書パス・PR番号）は、必ず前段の**完了報告から抽出した実値**を次段に渡す。推測値で渡さない。
- 実装は必ず worktree 内で行い、メインのチェックアウト（リポジトリ root）は `main` のまま保つ。フロー完了（マージ可）時に worktree を後片付けする。
- 各担当の責務境界を尊重する: planner は実装しない、developer は受け入れ判定しない、reviewer はコード修正しない。

## Agent Teams 版レビューフロー（実験）

上記「サブエージェント自動連携フロー」を基本とし、**review 段階だけ**を Claude Code の実験的機能 **Agent Teams**（複数の Claude インスタンスがチームとして協調）に置き換えるバリエーション。

- 採用理由: 公式が Agent Teams の入門に「research / review」を推奨しており、レビューは境界が明確・並列効果が大きい・ファイル競合が無く、Agent Teams の長所が最も素直に出る。逐次・単一成果物の planner / developer はサブエージェントのまま残す方が効率的（公式も逐次タスクはサブエージェント推奨）。
- 設計の出典: [docs/superpowers/specs/2026-06-14-agent-teams-review-design.md](docs/superpowers/specs/2026-06-14-agent-teams-review-design.md)
- 有効化: `.claude/settings.json` の `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1"` で ON。**この値を `"0"` にするか削除すれば、上記の純粋なサブエージェント運用に完全復帰する**（可逆）。表示は `teammateMode: "in-process"`（VSCode 統合ターミナルは split-pane 非対応のため）。
- 現状: **2026-06-15 に `"0"` へ戻して無効化済み**（純粋なサブエージェント運用に復帰）。下記の所見メモを参照。

### 所見メモ（2026-06-15 に無効化）

- **チームメイト同士が直接会話しなかった**ため、Agent Teams にした旨味をほとんど感じられなかった。設計（フロー手順3）が期待した「チームメイト同士の直接メッセージでの突き合わせ・相互チャレンジ」が実際には起きず、各 reviewer が独立に動いてリードが集約するだけになり、複数サブエージェントを並列起動するのと実質変わらない結果に。
- それでいてトークン消費は単一セッションより大幅増（各チームメイトが独立インスタンス）。コスト対効果が見合わないと判断し一旦無効化。
- 再挑戦するなら、チームメイト間の相互メッセージが実際に発生するよう spawn プロンプトで明示的に往復を強制する等の工夫が前提。それが無いなら基本フロー（逐次サブエージェント）で十分。

### フロー（基本との差分）

1. **planner を呼ぶ** — サブエージェント。現行どおり（差分なし）。
2. **developer を呼ぶ** — サブエージェント。現行どおり（差分なし）。
3. **review = チームを編成する**（ここが差分）
   - メイン Claude が**リード**となり、`.claude/agents/reviewer.md` を**チームメイト型として再利用**して複数の reviewer を起動する。各チームメイトには別レンズの spawn プロンプトを与える。例:
     - レンズA: 受け入れ条件の Playwright 動作確認（操作・スクショ証跡）
     - レンズB: `code-review` スキルでの差分のバグ／簡素化レビュー（読み取り専用、`--fix` 無し）
     - レンズC: 仕様適合・回帰・申し送り整合
   - チームメイト同士が**直接メッセージ**で指摘を突き合わせ（矛盾・重複の解消、相互チャレンジ）、リードが**1本の総合 PR コメント**に集約する。
   - 完了報告から「**総合判定（マージ可 / 要修正）**」を抽出する。
4. **判定による分岐** — 現行どおり。
   - **要修正** → 総合指摘を **developer（サブエージェント）** に渡して再実装させ、再度レビューチームに確認させる。**最大 3 回**まで。
5. **チームの後片付け** — レビュー完了後、**リードがチームを cleanup** する（`~/.claude/teams/`・`~/.claude/tasks/` は自動削除。cleanup はリードのみが実行）。判定が**マージ可**なら基本フロー同様 worktree も `git worktree remove` する。

### 注意（Agent Teams の制限）

- 実験的機能。トークン消費は単一セッションより大幅増（各チームメイトが独立インスタンス）。ルーチンな確認には基本フローを使う。
- `/resume`・`/rewind` で in-process チームメイトは復元されない → 中断したら作り直す。
- タスク完了マークがラグることがある／同時に1チームのみ／チームのネスト不可／リードは固定。
- 引き継ぎ情報（ブランチ名・worktree パス・仕様書パス・PR番号）は基本フロー同様、前段の完了報告から抽出した実値で渡す。レビューチームの各チームメイトも worktree 内で動作確認する。
